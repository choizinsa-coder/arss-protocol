# AIBA 거버넌스 2.0 — 설계 기준선 (S422 숙의 결과)

- 상태: **승인된 설계 기준선** (비오님, S422). 상세설계 착수 대기.
- 범위 한정: 본 문서는 **문서화** 산출물이며, 실제 시스템·코드·규정 변경이 아니다.
- 작성: 캐디 / 숙의 참여: 캐디 · 외부 도미 (비오님 중개) / 승인: 비오님(S422)

---

## 1. 배경 · 문제 정의

- 현행 변경 승인은 **직렬 사슬**이다: 도미 설계 → 비오 승인 → 제니 검증 → (구조 변경 시 제니 재검증) → 도미 최종 보증 → 실행 → 비오 최종 확인.
- 이 사슬에서 **도미(설계·최종 보증)와 제니(검증·재검증)가 둘 다 필수 통과 관문**이다. 게다가 "완료"의 정의 자체가 웬만한 작업에 제니 검증 통과를 요구한다.
- 따라서 한 에이전트만 멈춰도 전체가 정지한다("5분 코딩 55분 디버깅"의 구조적 원인).
- **목표(비오님 확정):** AI 개입을 줄이는 것이 아니라, 어떤 AI가 멈춰도 전체가 정지하지 않으면서 AI들이 최대 전문성으로 협업·반론·크로스체크하는 조직을 만든다.

---

## 2. 승인된 설계 원칙 17 (변경 기준선)

### 조직 철학
1. AI는 사실을 결정하지 않는다 — 관측·분석·제안까지. 결정은 규칙·자동검사·인간 승인이 한다.
2. AI는 경쟁자가 아니라 서로의 오류를 줄이는 동료다. **정신은 동료, 기능은 적대적 교차검증**.
3. 목표는 AI 축소가 아니라, 각 AI가 최대 전문성으로 협업·반론·크로스체크하는 조직.

### 역할과 사람의 분리
4. 역할(설계·구현·검증·감사)은 유지하고, 담당자는 교체 가능하다. 특정 AI가 필수 관문이 되지 않는다.
5. 담당자 자격은 자기 선언이 아니라 객관적 조건·증거로 판정한다.
6. 한 작업의 검증자는 그 작업의 설계자·구현자와 반드시 다른 주체여야 한다(자기 통과 금지).

### 위험 등급이 중심
7. 절차는 매번 사람이 정하지 않고, 변경의 위험 등급이 필요한 절차를 결정한다.
8. 위험 등급 판정 자체가 안전 급소다 — 애매하면 항상 높은 등급으로 본다.

### 멈추지 않되 안전은 지킴
9. 어떤 역할의 담당자가 없어도 전체가 정지하지 않는다. 대신 우아하게 축소한다(저위험 진행·기록, 고위험 대기).
10. "특정 AI가 없어서 막힘"과 "안전 조건을 아무도 못 채워 막힘"은 다르다. 후자의 고위험은 여전히 멈춘다(안전 바닥).
11. 축소 운영 중 한 일은 되돌릴 수 있는 것으로 제한하거나 복구 후 재검증하도록 표시한다.

### 협업의 규칙
12. 검증은 종착 관문보다 상시 관찰로 — 진짜 안전 위반일 때만 차단한다.
13. 토론에는 종료 조건이 있다: 합의 / 새 정보 없음 / 시간·비용 초과 / 비오 결정 요청. 과정은 토론이어도 결과는 하나의 분명한 사실로 기록한다.

### 인간 주권
14. 비오는 방향·철학·우선순위를 정하고 최종 승인한다. "전략만"은 기본값이지 권리 상실이 아니다.
15. 2단계 승인이라도 보안·거버넌스 구조 변경·되돌릴 수 없는 변경·비용 지출은 언제나 비오께 명시적으로 올라온다.

### 자동화의 자리
16. 자동화는 AI 대체가 아니라, 기계적 검사(형식·해시·테스트)를 맡겨 AI를 설계·검증·토론에 쓰기 위함이다. 자동 영역은 계속 넓히되, 자동검사도 "틀렸다"고 이의 제기할 통로를 남긴다.
17. 존재하지 않는 규모의 문제를 미리 설계하지 않는다. 현 3에이전트 현실에서 시작하고 필요할 때 확장한다.

## 3. 비오님 결정 (S422)

- **결정 1.** 위 17개 원칙을 거버넌스 2.0 상세설계의 **승인된 기준선**으로 채택한다. 단, 상세설계 과정에서 모순·누락·실행 불가능성이 확인되면 근거를 제시해 수정안을 올리고 비오님 승인을 받아 변경할 수 있다.
- **결정 2.** 제니 부재로 현행 정상 경로를 통한 재설계 착수가 막힌 교착을 인정한다. 거버넌스 2.0의 **설계·분석·문서화·검토까지만** 별도 경로 진행을 허가한다. 실제 시스템·코드·규정 변경은 별도 승인 전까지 금지한다.
- **결정 3.** 거버넌스 구조 재설계와 에이전트 운영 안정화를 **별도 트랙으로 병행**한다. 운영 안정화 트랙은 현재 장애의 **원인 확인과 최소 복구에 한정**하며, 새 거버넌스 구조를 선행 구현하거나 우회 적용해서는 안 된다.

## 4. 도미 상세설계 의뢰 사항 (대기 중)

**입력:** 위 17개 원칙. **산출:** 상세 아키텍처 + 안전 전환 절차.

**반드시 실제 작동까지 구체화할 7요소:**
1. **위험등급 판정** — 근거, 애매하면 높은 등급으로 실패(안전편향), 판정 주체와 위조·오조작 방지.
2. **역할 대체** — 역할 유지·담당자 교체. 자격의 부여·검증·회수 메커니즘. 검증자 ≠ 설계자·구현자(자기 통과 금지)를 구조로 강제.
3. **독립 검증** — 종착 관문이 아닌 상시 관찰, 안전 위반 시에만 차단. 결과는 하나의 분명한 사실로 기록.
4. **충돌 병합** — 병렬 검토자 상반 결론 시 규칙 기반 병합(예: 하나라도 안전 위반 지적 시 차단하는 안전-보수 병합). 통합자가 새 단일 관문이 되지 않게.
5. **토론 종료** — 종료 조건 세트 + 미합의 시 결론 규칙 + 비오께 의견 차이 정리해 올리는 탈출로.
6. **우아한 축소** — 담당자 전무 시 위험등급별 동작. 저위험 진행·기록, 고위험 대기. 안전 바닥 유지. 축소분은 되돌릴 수 있게 제한 또는 복구 후 재검증 표시.
7. **비오 승인 지점** — 필수 지점. 2단계 승인이라도 보안·거버넌스 변경·비가역 변경·비용 지출은 항상 비오께. "전략만"은 기본값이지 세부 개입·번복 권한 상실 아님.

**추가 포함:** 안전 전환 절차(제니 부재 교착에서 현 구조 → 2.0으로 안전 이행), 현 3에이전트 규모 반영(과설계 금지), 능력 기반 배정은 자기 주장이 아닌 실적·증거 기반.

**산출 형식:** 각 요소를 (a)작동 규칙 (b)실패·예외 처리 (c)원칙 매핑 3단 + 설계 이슈(원칙 모순·누락·실행불가) 별도 목록.

## 5. 현재 장애 상황 (S422 실측)

- 도미·제니 서비스는 기계 수준에서 **active**이나, 둘 다 **자기보호 차단(서킷브레이커)이 발동**해 요청을 거부한다("켜져 있지만 일은 거부"). 현재 가동 에이전트는 캐디뿐.
- S422에 위 4장 설계를 도미에게 의뢰 시도 → **도미 서킷브레이커로 실패**. 규칙에 따라 반복 호출·대리 설계 안 함.
- 이 사건 자체가 17개 원칙이 겨냥한 문제("한 AI가 멈추면 전체 정지")의 실증 사례다. 단, 결정 3에 따라 이를 핑계로 새 구조를 선행 적용하지 않는다.

## 6. 다음 세션 시작점

**우선순위: 운영 안정화 트랙 먼저** (설계 트랙이 물리적으로 막혔으므로)
1. **읽기전용 원인 확인:** 도미·제니 서킷브레이커 반복 트립의 공통 원인 조사(로그 등).
2. **최소 복구:** 런타임 재가동 등은 상태 변경 → **비오님 승인 필요**. 원인 확인 후 무엇을·왜 재가동하는지 정리해 상신.

**설계 트랙:** 에이전트가 하나라도 복구되면 → 도미에게 4장 설계 의뢰 재개.

**금지선:** 상태 변경은 승인 전 실행 금지 / 새 거버넌스 구조 선행 구현·우회 금지.
