# S439 — 비밀정보·비용 다중방어 구조: 설계 및 검증 기록

작성: 캐디 (S439) / 근거: 도미 설계 2회, 제니 검증 2회 (대화 내 원문 기반 정리)

## 1. 목표
AIBA의 모든 비밀정보(API Key, 토큰, 암호화 키)를 시스템 수준에서 보호하고, 어떤 사고(유출·폭주)에도 비용이 하드캡 이상 늘지 않는 구조를 설계한다. 이번 단계는 설계·검증만 수행하고 구현은 비오님 승인 후로 미룬다.

## 2. 캐디 실측 요약 (읽기 전용)
- 캐디 실행 런타임과 모든 서비스(mcp-bridge/jeni/domi/exec)가 전부 root(uid=0)로 구동.
- Grok 키는 repo .env에 존재(644->640으로 조임). 서버측 /etc/aiba/secrets.env에도 비밀 존재(다중 분산).
- Grok 키 값은 git 이력에 커밋된 적 없음(.env는 gitignore, 미커밋). 과거 외부유출 사유 없음.
- 중계 관문 없이 api.x.ai를 직접 호출하는 캐디 샌드박스 스크립트 10여 개 존재(우회면 큼).
- xAI 계정: 자동충전 OFF + 후불 미활성 -> 선불 소진 시 자동 거절(하드캡 적용, 비오님 완료).

## 3. 도미 설계 v1 (요약)
전용 Grok Relay(중계 서비스)를 신설하여 relay만 키를 쥐고 api.x.ai를 호출. 에이전트는 relay에만 요청. 서비스별 OS 계정 분리(root 해제), 예산 호출 전 선차단, nftables egress로 relay 계정만 외부 허용, 감사로그 키 마스킹, 직접호출 스크립트 정리. 8개 완료조건 시험 포함. 결정점: Bridge 통합(A) vs 별도 Relay 신설(B) -> B 선택(단일책임).

## 4. 제니 검증 v1 — STOP_SIGNAL ON
STOP 사유:
1. 캐디/서비스가 root인 한 secrets/토큰/relay메모리/egress가 전부 우회 가능(파일권한 무시, ptrace, 규칙 원복). loopback+토큰만으로는 경계 불성립.
2. 제안 relay 포트 8450이 기존 guardian veto 평면과 충돌.
3. 전환기 fallback(직접호출 유지)이 egress와 충돌해 우회창 생성.
4. 키 재발급이 필수인데 권고로만 처리됨(호스트 내부 노출 = 노출).
5. actor 토큰이 root-가독 파일이면 탈취 가능.
6. 완료조건을 캐디가 자기측정·자기선언하면 self-report 문제.
핵심: "캐디 root 해제 + 적용주체가 캐디 밖"이 완료 전제여야 함(OI-S435-002: 자기강등은 방어 아님).

## 5. 도미 설계 v2 (요약, 6건 해소 시도)
- root: de-root를 완료 전제로 승격, OS 계정 생성·systemd User= 변경·키 발급/revoke는 비오님 직접, 캐디는 검증만. 단 캐디 유닛은 uid=0 유지하되 일부 Cap 축소.
- 포트 8450->8451.
- fallback 금지 일괄 cutover(구키 revoke->egress 순서, 창 0 주장).
- 키 재발급 필수(신규 relay 전용 키 + 구키 revoke 완료게이트).
- actor 인증 systemd LoadCredential.
- 비밀 /etc/aiba/secrets.env 단일화 + 직접호출 스크립트 제거 하드게이트.
- 완료판정 7건 동시충족.

## 6. 제니 검증 v2 — STOP_SIGNAL ON (재차)
STOP 사유:
1. "캐디 uid=0 유지 + Cap 부분축소"는 경계 아님. root DAC로 secrets 직접 읽기, 유닛 원복으로 Cap 복원 가능. 제거목록에 DAC_OVERRIDE/SYS_ADMIN/FOWNER 부재.
2. LoadCredential은 env/journal 노출만 막고 root의 디스크·메모리 읽기는 못 막음. 조건1(에이전트가 원문 못 봄)에 캐디 root가 미포함.
3. revoke->egress 시간창 0 불성립: 신규키가 캐디 가독 위치에 있는 동안 직접호출 창 존재.
4. "캐디가 ansible 작성 + 비오 승인만"은 적용주체!=캐디 재위반.
5. 완료판정을 캐디가 측정·선언 = self-report.
재제출 최소조건: 캐디 실행경로 uid!=0을 예외없이 완료전제에 포함 / 비밀에 대한 캐디 비가독 명시 / ansible 대행 완화문구 삭제 / 신규키 직접경로 창 0 / 설계원문 파일 고정.

## 7. 최종 결론 (S439)
두 번의 검증 모두 같은 구조적 벽을 가리켰다: 진짜 병목은 캐디의 root 권한이며, 그 해제는 캐디가 아닌 손(비오님 또는 캐디 통제 밖 관리자)이 적용·검증해야 한다. 이는 설계 반복으로 풀리지 않는 전제이므로, 도미<->제니 설계 반복을 중단하고 비오님의 구조·OS 수준 결정(캐디 root 해제 방법·주체)을 다음 단계로 이관한다.

## 8. 이번 세션 확정·완료 조치
- xAI 비용 하드캡: 자동충전 OFF + 후불 미활성 (비오님 완료).
- repo .env / oauth_clients.json: 타 계정 읽기 차단(644->640).
- 설계·검증 원문 본 문서로 고정.
